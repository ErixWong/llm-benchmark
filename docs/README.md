## 测试目标

### 主要目标

1. **并发能力评估**: 确定API端点能够同时处理的最大请求数
2. **Token生成速度**: 测量LLM API的Token生成效率
3. **稳定性验证**: 验证API在高负载下的稳定性
4. **瓶颈识别**: 发现性能瓶颈和优化点

### 业务目标

- 确定系统容量规划
- 为SLA制定提供数据支持
- 优化资源配置
- 提升用户体验

## 测试指标

### 核心指标

| 指标名称 | 英文名称 | 说明 | 单位 |
|---------|---------|------|------|
| 吞吐量 | Throughput (RPS/QPS) | 每秒处理的请求数 | req/s |
| 响应时间 | Response Time (RT) | 请求从发出到收到响应的时间 | ms |
| 平均响应时间 | Average Response Time | 所有请求响应时间的平均值 | ms |
| P50响应时间 | P50 Latency | 50%请求的响应时间低于此值 | ms |
| P90响应时间 | P90 Latency | 90%请求的响应时间低于此值 | ms |
| P99响应时间 | P99 Latency | 99%请求的响应时间低于此值 | ms |
| 错误率 | Error Rate | 失败请求占总请求的比例 | % |
| 并发数 | Concurrency | 同时进行的请求数量 | 个 |

### LLM 特有指标

| 指标名称 | 英文名称 | 说明 | 单位 |
|---------|---------|------|------|
| Token生成速度 | Tokens Per Second (TPS) | 每秒生成的Token数量 | tokens/s |
| 首Token延迟 | Time To First Token (TTFT) | 从请求发出到收到第一个Token的时间 | ms |
| Token间延迟 | Token Inter-arrival Latency | 相邻Token之间的时间间隔 | ms |
| 输出Token数 | Output Tokens | 单次请求生成的Token数量 | 个 |
| 输入Token数 | Input Tokens | 单次请求的输入Token数量 | 个 |
| 总Token数 | Total Tokens | 输入+输出Token总数 | 个 |

### 资源指标

| 指标名称 | 说明 | 单位 |
|---------|------|------|
| CPU使用率 | 服务器CPU占用百分比 | % |
| 内存使用率 | 服务器内存占用百分比 | % |
| 网络带宽 | 网络数据传输速率 | MB/s |
| 连接数 | 活跃的网络连接数 | 个 |

## 测试类型

### 1. 负载测试 (Load Testing)

**目的**: 验证系统在预期负载下的性能表现

**参数**:
- 并发用户数: 逐步增加
- 持续时间: 10-30分钟
- 目标: 达到预期吞吐量

**示例配置**:
```javascript
{
  type: 'load',
  concurrency: [10, 50, 100, 200, 500],
  duration: '10m',
  rampUp: '2m'
}
```

### 2. 压力测试 (Stress Testing)

**目的**: 找出系统的性能极限和崩溃点

**参数**:
- 并发用户数: 持续增加直到系统崩溃
- 持续时间: 直到系统无法响应
- 目标: 确定最大承载能力

**示例配置**:
```javascript
{
  type: 'stress',
  concurrency: { start: 100, end: 2000, step: 100 },
  duration: '5m',
  rampUp: '30s'
}
```

### 3. 浸泡测试 (Soak Testing)

**目的**: 验证系统长时间运行的稳定性

**参数**:
- 并发用户数: 稳定负载
- 持续时间: 数小时到数天
- 目标: 检测内存泄漏、资源耗尽等问题

**示例配置**:
```javascript
{
  type: 'soak',
  concurrency: 100,
  duration: '24h',
  rampUp: '5m'
}
```

### 4. 峰值测试 (Spike Testing)

**目的**: 验证系统应对突发流量的能力

**参数**:
- 并发用户数: 突然增加和减少
- 持续时间: 短时间
- 目标: 观察系统恢复能力

**示例配置**:
```javascript
{
  type: 'spike',
  concurrency: [50, 500, 50, 500, 50],
  duration: '5m',
  rampUp: '10s'
}
```

### 5. Token速度测试 (Token Speed Testing)

**目的**: 专门测试LLM API的Token生成效率

**参数**:
- 输入Token数: 固定或变化
- 输出Token数: 固定或变化
- 并发数: 单线程或多线程

**示例配置**:
```javascript
{
  type: 'token-speed',
  inputTokens: [100, 500, 1000, 2000],
  maxOutputTokens: 500,
  concurrency: [1, 5, 10, 20],
  samples: 10
}
```

## 测试环境

### 硬件要求

| 组件 | 最低配置 | 推荐配置 |
|------|---------|---------|
| CPU | 4核 | 8核+ |
| 内存 | 8GB | 16GB+ |
| 网络 | 100Mbps | 1Gbps |
| 存储 | SSD | NVMe SSD |

### 软件要求

- Node.js >= 18.0.0
- npm >= 9.0.0
- 测试工具: autocannon, artillery, clinic

### 网络环境

- 与被测API的网络延迟 < 50ms
- 带宽充足，避免网络瓶颈
- 建议使用专用测试环境

## 结果分析

### 性能等级划分

| 等级 | P90响应时间 | 错误率 | 说明 |
|------|------------|--------|------|
| 优秀 | < 200ms | < 0.1% | 用户体验极佳 |
| 良好 | < 500ms | < 0.5% | 用户体验良好 |
| 一般 | < 1000ms | < 1% | 用户体验可接受 |
| 较差 | < 2000ms | < 2% | 需要优化 |
| 不可接受 | >= 2000ms | >= 2% | 必须优化 |

### LLM Token速度参考

| 模型类型 | TPS范围 | TTFT范围 | 说明 |
|---------|---------|----------|------|
| 小型模型 (7B) | 50-100+ | 100-300ms | 速度较快 |
| 中型模型 (13B-30B) | 30-60 | 200-500ms | 平衡性能 |
| 大型模型 (70B+) | 10-30 | 500-2000ms | 质量优先 |

## 最佳实践

### 测试前

1. **明确测试目标**: 确定要测试的指标和预期结果
2. **准备真实数据**: 使用生产环境相似的数据
3. **隔离测试环境**: 避免影响生产系统
4. **预热系统**: 测试前先运行预热请求

### 测试中

1. **监控系统状态**: 关注CPU、内存、网络等资源
2. **记录异常情况**: 详细记录错误和异常
3. **逐步加压**: 避免突然的大流量冲击
4. **多次测试**: 进行多次测试取平均值

### 测试后

1. **保存原始数据**: 保留所有测试原始数据
2. **对比历史结果**: 与之前的测试结果对比
3. **分析趋势**: 观察性能变化趋势
4. **持续优化**: 根据结果持续优化系统

### 常见陷阱

1. ❌ 测试数据过于简单，不能代表真实场景
2. ❌ 忽略网络延迟的影响
3. ❌ 测试时间过短，无法发现稳定性问题
4. ❌ 只关注平均值，忽略长尾延迟
5. ❌ 测试环境与生产环境差异过大
